Module 01 — The toolchain
IDE plugin or
harness?

Both have a place. Use the right one for the task. A harness gives you sub-agents, structured plans, and skill imports — but can be overkill. A plugin is fast for small tasks but missing features for complex ones.

IDE PLUGINS — simpler tasks
Copilot Cursor Windsurf built-in chat
HARNESSES — bigger features
Pi Claude Code OpenCode Codex

Personally: IDE plugin for simple tasks.
Pi to plan & work on new implementations.

Core principle — right tool, right job
"You can use a power drill to make a hole in a paper — but it's probably overkill."

If you need to write a for-loop, you don't need Opus or GPT-5.5. If you need to implement a method from a signature, GPT-mini will get the job done. Match model capability to task complexity.

for-loop → mini method impl → Sonnet architecture plan → Opus code review → GPT-5.5

LIGHTER MODEL

Module 03 — Context hygiene
Keep context in the first half

As context grows, the model tracks more variables — and hallucinations increase. Stay in the first half of the window for best results.

✓ Export progress to documents between sessions
✓ Delegate search tasks to sub-agents
✓ Start fresh sessions often — load the plan file, not memory
IDEAL — stay here
✓ your work
still ok
danger zone
First half — best results
Middle — caution
Full — start fresh
TECHNIQUE — frequent intentional compaction
Deliberately structure how you feed context throughout development. Export a progress doc, open a new session, load the doc. Don't keep accumulating.
Module 04 — Requirements
The grill-me skill

Before writing a single line of code, let the AI ask you questions about the feature. This builds shared context and surfaces assumptions you didn't know you were making.

Some questions you can answer immediately. Others go back to your PM. Either way — you catch gaps before they become bugs.

WORKFLOW
1 · Paste ticket into harness
2 · Run grill-me skill
3 · Answer what you can
4 · Escalate the rest to PM
5 · Export plan with acceptance criteria
6 · Pass plan to implementation session

The AI is trying to reach a common understanding with you — let it.

Module 05 — Testing
No one wants to write tests.
The AI will.
01 — GENERATE
Tests from the plan
Once you have a concrete plan and acceptance criteria, ask the AI to generate tests before you start implementing. One prompt, done.
02 — RUN TDD
Tests first, agent implements
Give the agent the failing tests. Let it implement until they pass. This is the most reliable autonomous workflow available today.
03 — FUTURE SESSIONS
Tests as a contract
Tests tell future sessions — and future developers — how the code is supposed to behave. They're not just CI — they're documentation.
Core principles
What separates
the devs who succeed

It isn't the tool. It's the process. The developers who get the most out of AI are the ones who fall back on engineering fundamentals.

01
Plans are first-class artifacts
Export before implementing. Pass across sessions.
02
Models can't check their own work
Always review in a fresh session with a capable model.
03
Context size is a dial, not a given
Compact deliberately. Export often. Start fresh.
04
Fundamentals matter more than ever
Vertical slices, TDD, ubiquitous language — not relics.
05
Be token-aware, not token-paranoid
Right model, right task, rational context = you're fine.
